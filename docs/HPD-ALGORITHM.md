# HPD: Hierarchical Probabilistic Decomposition

[![DOI](https://zenodo.org/badge/DOI/10.5281/zenodo.20366383.svg)](https://doi.org/10.5281/zenodo.20366383)

*A formal description of the algorithm behind LXPEN.*

## Abstract

Hierarchical Probabilistic Decomposition (HPD) is a password candidate generation algorithm that models human password creation as a structured probabilistic process. Instead of enumerating passwords from external wordlists or brute-forcing character spaces, HPD decomposes the password space into a hierarchy of **structural patterns**, **semantic slots**, and **frequency-weighted entries**. This produces a compact candidate space (~4.3M) that covers ~90% of pattern-based human passwords while requiring zero external data.

## 1. Problem Statement

Given an NTLM hash `H = MD4(UTF16LE(password))`, find the plaintext `password`.

Traditional approaches:
- **Brute force**: Enumerate all strings of length 1..N. Space grows exponentially: `95^8 ≈ 6.6 × 10^15` for printable ASCII up to 8 characters. Infeasible on CPU.
- **Dictionary attack**: Use a precompiled wordlist. Coverage depends entirely on list quality. A 100K wordlist with best64 rules generates ~6.4M candidates but misses passwords not derivable from wordlist entries.
- **Rule-based**: Apply transformation rules (capitalize, l33t, append digits) to wordlist entries. Multiplicative expansion but still limited by base wordlist coverage.

**HPD insight**: Most human-chosen passwords are not random strings. They are compositions of a small number of semantic building blocks arranged in predictable structural patterns. Model the composition process directly.

## 2. Formal Model

### 2.1 Definitions

- **Slot Type** `T`: A semantic category of password components.
  - `T ∈ {LowerWord, UpperWord, CapWord, Name, CapName, SeqDigits, Year, Symbol, Keyboard, L33t, CapL33t, Literal}`

- **Slot Entry** `e = (value, frequency)`: A concrete string value with an estimated probability within its slot type.
  - Example: `("password", 0.032) ∈ LowerWord`

- **Slot** `S = (type, fixed_value?)`: A position in a pattern that draws from a slot type, or contains a fixed literal.

- **Pattern** `P = (name, [S₁, S₂, ..., Sₖ], frequency)`: A structural template with k slots and an estimated frequency of occurrence in real-world passwords.
  - Example: `("lower_digits", [LowerWord, SeqDigits], 0.224)`

- **Candidate** `c = concat(e₁.value, e₂.value, ..., eₖ.value)`: A password string generated by selecting one entry from each slot.

### 2.2 Candidate Space

For a single pattern `P` with slots `[S₁, ..., Sₖ]`:

```
Candidates(P) = |entries(S₁)| × |entries(S₂)| × ... × |entries(Sₖ)|
```

Total candidate space:

```
Total = Σ over all patterns P: Candidates(P)
```

With 45 patterns and current slot data: **Total = 4,274,914**

### 2.3 Probability Model

The probability of generating candidate `c` from pattern `P` with entries `e₁, ..., eₖ`:

```
P(c | P) = Π(i=1..k) freq(eᵢ) / Σ(entries in Sᵢ) freq(e)
P(c) = freq(P) × P(c | P)
```

Where `freq(P)` is the pattern's estimated frequency and `freq(eᵢ)` is the entry's frequency weight.

### 2.4 Ordering

Candidates are generated in approximate probability order:
1. Patterns sorted by descending `freq(P)`
2. Within each pattern, slot entries sorted by descending `freq(e)`
3. Cartesian product enumerated in row-major order (leftmost slot varies slowest)

This ensures the most likely passwords are tested first, maximizing the probability of early exit.

## 3. Pattern Design

### 3.1 Pattern Taxonomy

Patterns are derived from empirical analysis of leaked password databases and password creation psychology:

**Single-component patterns** (5 patterns):
```
[LowerWord]           "password", "dragon", "butterfly"
[SeqDigits]           "123456", "000000"  
[Keyboard]            "qwerty", "asdfgh"
[Name]                "michael", "jessica"
[CapWord]             "Master", "Shadow"
```

**Two-component patterns** (22 patterns):
```
[LowerWord + SeqDigits]    "shadow99", "dragon69"
[CapWord + SeqDigits]      "Admin123", "Master007"
[Name + Year]              "michael1994", "jessica2024"
[LowerWord + Year]         "summer2024", "love1994"
[L33t + SeqDigits]         "h4ck3r666", "p@ss123"
[SeqDigits + LowerWord]    "123password" (reversed)
...
```

**Three-component patterns** (18 patterns):
```
[LowerWord + SeqDigits + Symbol]     "password123!"
[CapName + Year + Symbol]            "Michael1994!"
[Name + Year + Symbol]               "michael1994."
[L33t + SeqDigits + Symbol]          "h4ck3r666!"
[Symbol + CapWord + SeqDigits]       "!Admin123" (prefix symbol)
...
```

### 3.2 Frequency Assignment

Pattern frequencies are estimated from:
- Published academic research on password composition (Weir et al., 2009; Veras et al., 2014)
- Aggregate statistics from publicly analyzed breach datasets
- Empirical validation against test sets

The top 5 patterns by frequency account for ~60% of all human-chosen passwords:
1. `lower_digits` (22.4%) — The dominant pattern
2. `cap_digits` (14.1%)
3. `lower_only` (11.8%)
4. `lower_year` (8.9%)
5. `name_year` (7.2%)

## 4. Slot Data Design

### 4.1 Data Sources

Slot entries are curated from:
- Published frequency lists of common password components
- Cultural/linguistic analysis (currently English + Turkish)
- Domain-specific knowledge (football teams, l33t conventions)

### 4.2 Slot Type Details

| Type | Entries | Coverage Strategy |
|---|---|---|
| LowerWord | 190 | Top common passwords, English words, Turkish words, sports, tech, gaming |
| Name | 122 | EN male (38) + EN female (20) + TR male (38) + TR female (26) |
| Year | 61 | 1970-2026, plus culturally significant years (1453, 1907, 1923) |
| SeqDigits | 62 | Single digits through 9-digit sequences, common patterns (007, 420, 666, 786) |
| Symbol | 25 | Single and multi-char symbols, ordered by usage frequency |
| L33t | 28 | Common l33t transformations of popular words |
| Keyboard | 24 | QWERTY keyboard walks and repeating patterns |

### 4.3 Derived Slot Types

Some slot types are derived from base data via transformation:

- `CapWord` = `LowerWord.map(capitalize)` — "password" → "Password"
- `UpperWord` = `LowerWord.map(upcase)` — "password" → "PASSWORD"
- `CapName` = `Name.map(capitalize)` — "michael" → "Michael"
- `CapL33t` = `L33t.map(capitalize)` — "p@ssw0rd" → "P@ssw0rd"

This avoids data duplication while tripling the effective vocabulary.

## 5. Implementation

### 5.1 C-Native Generation

The critical insight for performance: candidates are generated and hashed entirely in C, eliminating interpreter overhead.

```c
// Per-thread worker (simplified)
for (size_t k = start; k < end; k++) {
    // 1. Decode combination index k → slot indices
    // 2. Concatenate slot values → candidate string
    // 3. Convert to UTF-16LE
    // 4. MD4 hash
    // 5. Compare against all active targets
}
```

The cartesian product is indexed numerically:
```
k = 0, 1, 2, ..., (total_combinations - 1)

slot_index[i] = (k / divisor[i]) % slot_count[i]

where divisor[i] = Π(j=i+1..num_slots-1) slot_count[j]
```

This allows any thread to compute any combination independently — perfect for data-parallel execution.

### 5.2 Multi-Target Optimization

When cracking N hashes simultaneously:
- Each candidate is hashed **once** (the expensive operation)
- The resulting hash is compared against **all N active targets** (cheap 16-byte comparison)
- When a target is found, it's deactivated via atomic CAS — no lock contention
- When all targets are found, all threads exit via atomic flag

Cost model:
```
Single target:   O(C × H)           where C = candidates, H = hash_time
Multi-target:    O(C × H + C × N × cmp_time)
                ≈ O(C × H)           since cmp_time << H for small N
```

Cracking 20 hashes costs essentially the same as cracking 1.

### 5.3 Thread Model

```
lxpen_crack_pattern() called per pattern:
  ├─ Calculate total_combinations = Π slot_counts
  ├─ Precompute divisors[] and slot_offsets[]
  ├─ Spawn N threads (N = CPU cores)
  │   ├─ Thread 0: combinations [0, chunk)
  │   ├─ Thread 1: combinations [chunk, 2*chunk)
  │   └─ ...
  ├─ Each thread:
  │   ├─ For each combination in range:
  │   │   ├─ Check all_done flag (atomic)
  │   │   ├─ Decode combination → slot indices
  │   │   ├─ Build candidate from slot values
  │   │   ├─ UTF-16LE convert + MD4 hash
  │   │   └─ Compare vs active targets (atomic CAS on match)
  │   └─ Return (newly_found, tried)
  ├─ Join all threads
  └─ Return total_found
```

## 6. Comparison with Related Work

### 6.1 vs Dictionary Attack (Hashcat/John)

| Aspect | Dictionary + Rules | HPD |
|---|---|---|
| Coverage source | External wordlist | Internal pattern model |
| Coverage limit | Words in list × rule expansion | All combinations of slot entries |
| Name+Year passwords | Poor (need name in wordlist + year rule) | Native support |
| L33t passwords | Rule-dependent | Native L33t slot entries |
| Cultural passwords | Need culturally-specific wordlist | Slot data includes cultural entries |
| Disk I/O | Required (wordlist read) | Zero |
| RAM | Wordlist loaded in memory | Slot tables only (~KB) |

### 6.2 vs PCFG (Probabilistic Context-Free Grammar)

Weir et al. (2009) proposed PCFG-based password cracking. Comparison:

| Aspect | PCFG | HPD |
|---|---|---|
| Training | Requires training corpus | No training (curated slot data) |
| Grammar | Learned from data | Manually designed patterns |
| Probability model | Formal grammar probabilities | Pattern frequency × slot frequency |
| Candidate ordering | Probability-ordered | Approximate probability order |
| Extensibility | Retrain on new data | Add entries to slot tables |

HPD can be viewed as a hand-crafted PCFG with domain-expert-designed production rules and frequency weights.

### 6.3 vs Markov Models

| Aspect | Markov Chain | HPD |
|---|---|---|
| Unit | Character n-grams | Semantic components |
| Training | Requires corpus | No training |
| Candidates | Character-by-character generation | Pattern-by-pattern generation |
| Semantic awareness | None (statistical only) | Full (name, word, year, l33t) |

## 7. Limitations

1. **Random passwords**: HPD cannot crack truly random passwords (e.g., `xK9#mZ2pLq`). No pattern-based approach can.
2. **Rare patterns**: Passwords following unusual structures not in the 45 patterns are missed.
3. **Long passwords**: Current slot data focuses on components up to ~15 characters. Passphrases (4+ words) need additional patterns.
4. **Non-Latin scripts**: Current data covers English and Turkish. Other languages need slot data additions.
5. **NTLM only**: Current hash function is NTLM. Other hash types require implementation.
6. **No salt handling**: NTLM is unsalted. Salted hashes (bcrypt, scrypt) would require per-salt candidate generation, reducing the multi-target advantage.

## 8. Future Directions

- **Adaptive frequency learning**: Adjust slot frequencies based on which candidates match during a session
- **Markov-enhanced ordering**: Use character-level Markov chains within slots for finer probability ordering
- **GPU acceleration**: Port the hash+compare inner loop to OpenCL/CUDA
- **Passphrase patterns**: Add multi-word patterns for longer passwords
- **Hybrid mode**: Use HPD patterns first, then fall back to a small wordlist for remaining targets
